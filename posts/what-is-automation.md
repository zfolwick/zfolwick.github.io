# What is test automation? What is it not?

Test Automation is the act of taking a recording of the understanding of a system under test at a moment in time, and exercising that recording, and confirming the expected outcome.
## What is UI test automation?
UI Test Automation is more specific: It’s a recording of the understanding of the steps and state a user interface appears to undergo in order to exercise a system under test at a moment in time; the automation framework then performs those steps, with assertions confirming that understanding is still correct.

Test automation isn’t about creating a robot to exercise an application or coding styles or patterns- those are simply a means to an end.  For the software industry, the end result is the communication of how closely an application matches the set of expectations created by stakeholders; more importantly, well-crafted test automation informs users about failures of the application to conform to expected results.

UI test automation frameworks all suffer from the same issues:
1. Because they often simulate user actions, it’s thought that they can simply stand-in for users.
2. They cannot tell why they failed from a business perspective, only what the test automation code couldn’t accomplish.
3. They are built on the tail-end of a chain of technologies- any one of which might fail at any point, causing what might be viewed as UI bugs.

Both of these cause ongoing symptoms that give test automation a reputation as notoriously difficult to extract value from.  That’s why I recommend UI test automation proceed as part of a system of quality checks.

UI test automation is sometimes approached as a way to automate manual qa efforts, essentially “doing qa work faster”. This belies a lack of understanding of the value that manual QA can provide. Quality Assurance is thinking and accountability wrapped into an actionable report. It’s not simply a box to check, it’s the verification that your team’s efforts conform to (a) the user’s expectations, or (b) the stakeholder’s requirements.

UI test automation simulating user’s actions does not mean that a project director should consider automation as a replacement for manual QA. This is an insult to QA, as well as misses the mark of the benefits of UI automation. 

## Antipatterns: Confidently killing progress
Antipatterns in software development are programming implementation patterns that describe common ways in which people do the Wrong Thing. It might be using GOTOs (I'm not even sure which languages still even _allow_ goto statements anymore), bad naming schemes, god-functions, and global variables. It's not limited to just coding- it's 2026 and we live in bizarro-world. People mess up everywhere, and it shouldn't be a surprise that business assumptions cause major issues. Below are two million-dollar anti-patterns: business practices that, when followed, can cause a million dollars of churn- wasted time fixing. Antipatterns are present because businesses think that's the best way to do business, but a software professional knows that can sometimes be completely wrong.

### Antipattern: Automation replacing manual QA
Because they simulate user actions by interacting directly with the DOM, UI test automation results are often interpreted as user-level defects. In practice, the primary role of UI automation in a developer’s hands is to verify that what the application was instructed to render is what actually appeared on the page—validating contracts between code and what the user sees. In doing so, automation frequently uncovers unexpected DOM artifacts, an abstraction that developers in modern UI frameworks are largely divorced from. These findings are valuable for detecting rendering mismatches or integration regressions, but they are not necessarily representative of user-visible bugs. Automated walkthroughs of user steps are secondary; confusing DOM-level validation with true user experience testing leads to low-value failures, risking being labelled as “brittle tests” and misclassifying failures.

UI test automation cannot explain why a failure matters from a business perspective, only what the automation code was unable to do. A failed selector, timing issue, or missing DOM node provides a technical symptom, not intent, impact, or user relevance. Automation reports the mechanics of failure: assertions unmet, actions blocked; but lacks the context to determine whether the outcome violates a business rule, a user expectation, or a stakeholder requirement.  This is automation pointing inwards- masking the issue with the app simply because out of the box, automation can’t know about an app. Without human interpretation, these failures risk being treated as defects of equal weight, obscuring real risk and reducing test results to operational noise rather than actionable insight.

### Antipattern: All the testing eggs in a single basket
UI automation operates at the end of a long chain of technologies, which makes failures difficult to interpret when upstream data sources are treated as opaque. When UI tests don’t validate upstream API responses, failures force manual investigation and still can’t clearly answer whether the UI is broken or the data feeding it is wrong. Rendering issues that surface in the browser are often symptoms of malformed, incomplete, or contract‑breaking API responses rather than true presentation defects. By combining UI validation with direct inspection of API response content, automation can explain why the UI behaved as it did, separating UI failures from data and integration issues and turning misleading UI noise into traceable, system‑level signals.

## The Fix: Back to QA Fundamentals
One aspect of test automation is the rapid, repeated, and stable expression of expectations of a system under test.  When the system under test fails these expectations, traditionally, test automation expresses this failure as the automation framework being incapable of acquiring the next expected locator.  This makes it look like a failure of the automation code when in fact, the automation framework could tell you something about the system under test, if the signals were fixed.

### Fix 1: Don’t assume critical steps complete successfully
Test automation is ultimately software engineering, which is why test automation experts are called SDETs.  Automation scripts are steps described to the point of tedium. Not only do automation engineers need to understand software engineering, but they also need to be able to write software that handles unmet expectations and unexpected results at any point. When a step in a test script fails, if it’s critical, the test harness should get an error pointing to the step that failed and not- as is common in test automation frameworks- the next step as the failure. 

When building test automation frameworks, critical steps that display symptoms of failure need to assert successful state immediately after.  Failures of execution must be obvious in where the execution step failed.

### Fix 2: Repro steps are the goal, not explaining the failure
If automation is being asked by project managers and devs why a test failed, as long as the test started its life as a validly passing test, the interrogation must be answered with “these are the steps that were agreed upon; if the steps led to a failure, it’s a failure in some decision upstream of the tests- either a failure of project management, qa, or development to notify required changes to test automation, or a failure of either the frontend application, a backend issue, or a network issue”; e.g., the problem is either a problem between business communication & processes, or an actual regression in the application. The above reply puts the onus on developers to debug the issue, and that only works with reliable, stable, error reproduction.

In environments where there’s no guarantee that the network or application can deliver stable results, UI automation frameworks must observe some information about network ability and network communication.  This is vital in pre-production environments plagued by “not my problem”-based information silos, because UI automation in these testing environments might actually be the first time an api is being actually tested! Upstream teams willing to put out untested code bring with them a special hubris willing to push back on any bug potentially attributable to them, so any UI automation failures due to network or api deficiencies must contain a stable reproduction of the error that points directly to the upstream dependency failing. UI test observability of the API in these instances is mission-critical to these kinds of high-risk projects.

### Fix 3: Find out who is responsible for particular steps completing, all the way to the database layer
Combining the first two fixes above leads to one inevitable outcome: accountability. In today’s application environment, services collaborate to serve the needs of the customer. When teams responsible for these services fail at that mission, they need to be informed ASAP. If you’ve taken the above two fixes seriously, you’ve implemented test scripts that look to have an accountable party (or parties) at each individual step, and reproducible steps to not only convince sometimes reluctant parties they need to act, but also give them steps to consistently put out a provably high quality product. These automation and QA frameworks know exactly who they need to reach out to with a bug, without creating churn, because the responsible party is written into the test framework, and encoded in the test automation assertions themselves.

# So what's it all for then?
Software test automation _can not_ replace manual QA work- it can only automate sending signals about application state. Automation is a recording of decisions that were made- it always includes the steps taken to complete a task, but it _should_ include attribution of failed steps to responsible teams in automation reports. A well-trained quality manual QA Engineer should be able to take good automation results and reproduce the error, but the automation should record the steps taken and the responsible teams, making manual QA work easier to scale, while increasing quality.
